# pnpm Workspace Conversion

**Status:** Draft
**Created:** 2026-04-27

## Goal

Convert the project to a pnpm workspace as a focused infrastructure PR before shared-LS extraction begins ([2026-04-16_shared-language-services.md](2026-04-16_shared-language-services.md)). Validates the workspace setup in isolation — vsce, CI, release.yml, and vscode-dvala's install path — so that shared-LS extraction inherits a stable workspace foundation rather than mixing two infrastructure changes.

Single PR. Lands after the pnpm migration ([2026-04-26_pnpm-tooling-refresh.md](2026-04-26_pnpm-tooling-refresh.md)) is on `main`. Resolves issue [#117](https://github.com/mojir/dvala/issues/117) as a side effect.

---

## Background

### Current state (post-pnpm-migration)

Two `package.json` files, two lockfiles, two install paths:

- Root `package.json` covers `src/`, `cli/`, `playground-www/`, `mcp-server/`, `playground-builder/`, `common/` — everything except the VS Code extension.
- [vscode-dvala/package.json](vscode-dvala/package.json) is independent — its own deps (`@vscode/debugadapter`, `@vscode/debugprotocol`, `@types/vscode`, `@types/node`, `typescript`).
- Root `pnpm-lock.yaml` and `vscode-dvala/pnpm-lock.yaml` are separate. Each is regenerated by its own `pnpm install`.
- The wireit `typecheck` task runs `pnpm install --dir vscode-dvala` to make sure the sub-package's deps are present before typechecking.
- Version sync between the two `package.json` files is handled by **three** separate code paths (issue #117): the build script mutates the tracked file, `release.yml` has an inline `node -e` script, manual edits go undetected.

### Why convert now (vs. later)

Shared-LS extraction is the next major piece of work. Its plan calls for `src/shared/` modules that need to be importable by both vscode-dvala (Node) and playground-www (browser). That extraction is most cleanly modeled as a workspace member: vscode-dvala depends on it via `workspace:*`, playground-www imports it directly, and pnpm symlinks resolve it consistently.

Doing workspace conversion **as part of** shared-LS extraction means one PR does double duty: layout change *and* code extraction. Bisect signals get muddy if anything regresses. Doing it **before** shared-LS extraction means the workspace setup is validated in isolation:

- Does `vsce package --no-dependencies` still produce a working `.vsix` under symlinked `node_modules`?
- Does `pnpm install --frozen-lockfile` in CI cover all workspace members?
- Does the release version-bump still work cleanly?

Each of those is a workspace-specific question; none of them depend on shared-LS code being extracted. Validate them on the existing structure first, then layer extraction on top.

### What we get

1. **Single lockfile.** Root `pnpm-lock.yaml` covers vscode-dvala. The sub-package lockfile goes away.
2. **Single install command.** `pnpm install` at the root installs everything; the wireit `typecheck` task no longer needs its own install step.
3. **Native version propagation.** `pnpm -r version <bump>` bumps all workspace members consistently. Replaces the inline `node -e` script in `release.yml` and the build-time mutation in `vscode-dvala/build.mjs`. Resolves issue #117.
4. **Dep dedup.** vscode-dvala's `@types/node`, `typescript`, etc. share with root via pnpm's content-addressed store and symlinks.
5. **Foundation for shared-LS.** New shared modules can be declared as workspace members and consumed by both vscode-dvala and playground-www without further infrastructure changes.

### What we don't get (yet)

This conversion adds **one** workspace member: `vscode-dvala`. Other directories that *could* eventually become members (`cli/`, `mcp-server/`, `playground-www/`, `playground-builder/`, `common/`) stay under the root `package.json` for now. They don't have their own `package.json` files, and creating them is a separate scope decision tied to shared-LS extraction (which will introduce concrete reasons for some of them to become members).

## Proposal

### 1. Declare the workspace

Add `pnpm-workspace.yaml` at the repo root:

```yaml
packages:
  - vscode-dvala
```

Single member for now. Future members get added as they emerge (likely starting with the shared-LS extraction).

### 2. Remove the sub-lockfile

Delete `vscode-dvala/pnpm-lock.yaml`. Root `pnpm-lock.yaml` becomes the single source of truth for the workspace.

### 3. Update the wireit `typecheck` task

Currently ([package.json:221](package.json#L221)):

```jsonc
"typecheck": {
  "command": "pnpm install --dir vscode-dvala && tsc -p ./tsconfig.compile.json --noEmit && tsc -p vscode-dvala/tsconfig.json --noEmit",
  ...
}
```

Becomes:

```jsonc
"typecheck": {
  "command": "tsc -p ./tsconfig.compile.json --noEmit && tsc -p vscode-dvala/tsconfig.json --noEmit",
  ...
}
```

Drop the install step — root `pnpm install` already covers vscode-dvala under the workspace.

### 4. Replace the version-sync code paths

[release.yml:42-49](.github/workflows/release.yml#L42) currently does:

```yaml
NEW_VERSION=$(node -p "require('./package.json').version")
echo "version=$NEW_VERSION" >> "$GITHUB_OUTPUT"
node -e "
  const fs = require('fs');
  ...
  for (const subPkg of ['vscode-dvala/package.json']) {
    const pkg = JSON.parse(fs.readFileSync(subPkg, 'utf-8'));
    pkg.version = version;
    ...
  }
"
```

With workspaces, replace the whole block with:

```yaml
pnpm -r version ${{ inputs.bump }} --no-git-tag-version
NEW_VERSION=$(node -p "require('./package.json').version")
echo "version=$NEW_VERSION" >> "$GITHUB_OUTPUT"
```

`pnpm -r version` (recursive) applies the bump to every workspace member's `package.json`. Single source of truth, single command.

Also remove the build-time mutation at [vscode-dvala/build.mjs:9-13](vscode-dvala/build.mjs#L9-L13):

```js
// REMOVE — workspace-aware version sync handles this at release time
if (extPkg.version !== rootPkg.version) {
  extPkg.version = rootPkg.version
  writeFileSync(extPkgPath, ...)
}
```

The extension's `package.json` becomes a normal tracked file — the workspace's release-time bump keeps it in sync.

**Lockfile side effect of `pnpm -r version`.** Unlike a non-workspace setup, sub-package versions *do* appear in the root `pnpm-lock.yaml` (in the `importers.<path>` block). So `pnpm -r version` will modify the lockfile. The release workflow needs to either:

- Run `pnpm install --lockfile-only` after the version bump to refresh the lockfile, then commit both, or
- Accept that `pnpm -r version` updates the lockfile in-place and commit the result directly.

Either works; lockfile-only is more explicit and survives future pnpm-version changes better. Recommended.

### 5. Verify vsce still works

Highest-risk surface. The current `vsce package --no-dependencies` invocation at [vscode-dvala/build.mjs:39](vscode-dvala/build.mjs#L39) needs to:

- Still resolve `../node_modules/.bin/vsce` correctly under workspace-hoisted layout.
- Still produce a working `.vsix` (vscode-dvala's own deps are now symlinked under workspace `node_modules`, but `--no-dependencies` skips dep validation so this should be fine).

Smoke test: build the `.vsix`, install in VS Code, confirm the extension activates and the debugger works.

**Fallback chain on vsce trouble** (in order):

1. Add `node-linker=hoisted` to root `.npmrc` if vsce trips on symlinked `node_modules`. Loses pnpm's strict-layout phantom-dep detection but matches npm's flat layout.
2. Set `shared-workspace-lockfile=true` (default) and `link-workspace-packages=true` in `.npmrc`.
3. As a last resort, keep vscode-dvala out of the workspace (revert this PR) and accept the dual-lockfile setup as long-term state.

### 6. Update CI workflows

The four workflows in `.github/workflows/` already use `pnpm install --frozen-lockfile`. Under workspaces, that single command covers all members — no other change needed. The cache key on `setup-node`'s `cache: 'pnpm'` automatically follows the lockfile path.

### 7. Update CLAUDE.md

Add a short note under "Project Structure" or a new "Workspace Layout" section:

- Root is a pnpm workspace; `vscode-dvala/` is the (currently sole) workspace member.
- `pnpm install` at the root installs everything; no need to install in sub-directories.
- Versions are bumped via `pnpm -r version` in the release workflow; never manually edit version fields.

## Open Questions

- **vsce + workspace `node_modules` layout.** The smoke test will tell us. If it fails, fallback to `node-linker=hoisted` (section 5). Worth a 30-min spike before committing to the conversion.
- **Should this PR also create `package.json` files for `cli/`, `mcp-server/`, etc.?** No — they don't have separate publish surfaces, separate dep sets, or separate versioning needs *today*. Adding them now is over-engineering. The shared-LS extraction will drive concrete reasons for some of them to become members; defer until then.
- **Do we want `pnpm -r version` or a custom script?** `pnpm -r` is the standard idiom and replaces ~10 lines of inline node-e. If we ever need finer control (e.g. only bump specific members), we can add a script later.
- **Will Renovate's lockfile maintenance still work?** Yes — Renovate understands workspace lockfiles (it has explicit support for pnpm workspaces). One root lockfile is actually friendlier to Renovate than two separate ones.

## Implementation Plan

1. **Spike vsce + workspaces locally.** Before doing any commits, create a throwaway branch, add `pnpm-workspace.yaml`, run `pnpm install`, and try `pnpm run build-vscode-ext`. If the `.vsix` builds cleanly and works in VS Code, proceed. If not, evaluate the fallback chain (section 5) before committing to the design.
2. **Add `pnpm-workspace.yaml`** with `vscode-dvala` as the only member.
3. **Delete `vscode-dvala/pnpm-lock.yaml`.** Root install will own vscode-dvala's deps.
4. **Run `pnpm install`** to regenerate the root lockfile with workspace-aware resolution. Inspect the diff: vscode-dvala's deps should now appear under `importers.vscode-dvala` in the root lockfile.
5. **Update [package.json:221](package.json#L221)'s wireit typecheck task** to drop `pnpm install --dir vscode-dvala`.
6. **Update [release.yml:42-49](.github/workflows/release.yml#L42)** to use `pnpm -r version <bump> --no-git-tag-version` followed by `pnpm install --lockfile-only` (to refresh the lockfile cleanly).
7. **Remove the build-time mutation** at [vscode-dvala/build.mjs:9-13](vscode-dvala/build.mjs#L9-L13). The build script becomes a pure builder again.
8. **Smoke test:**
   - `pnpm run check:no-fix` — full pipeline.
   - `pnpm run build-vscode-ext` — verify the `.vsix` is produced.
   - Install the `.vsix` in VS Code locally — verify it activates and the debugger works.
   - Trial `pnpm version patch --no-git-tag-version` (then revert) — confirm the lockfile-only refresh keeps things consistent.
9. **Update [CLAUDE.md](CLAUDE.md)** with the workspace note (section 7).
10. **Close issue [#117](https://github.com/mojir/dvala/issues/117)** as resolved by this PR — the version sync is now structurally enforced by `pnpm -r version`, not by drift-prone manual paths.

## Risks

- **vsce edge cases** — main unknown. Mitigated by the spike in step 1 and the fallback chain in section 5.
- **Renovate / Dependabot adjustment** — if Renovate is already running by the time this lands, its config may reference the old lockfile structure. Verify after merge.
- **Contributor confusion** — anyone with a stale checkout will see weird `node_modules` symlinks. CLAUDE.md note + the natural "delete and reinstall" reflex covers this.

---

## Out of scope

- **Restructuring `src/` into sub-packages.** Owned by shared-LS extraction.
- **Adding more workspace members beyond vscode-dvala.** Adds churn without benefit until there's a concrete reason. Wait for shared-LS to surface real candidates.
- **Changing the publish flow.** `npm publish` still publishes the root package; vscode-dvala has its own marketplace publish path via vsce. Workspaces don't change either.
- **Workspace catalogs** (pnpm's centralized version pinning across members). Useful for >5 members; over-engineered for two.

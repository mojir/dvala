# Bundler as a Macro

## Vision

The bundler is a macro — AST in, transformed AST out. It uses the same effect system and macro infrastructure as everything else in Dvala. No separate plugin API, no configuration language.

## Core Pipeline

1. Walks the AST, finds `import()` → performs `@import.resolve` → inlines module AST
2. Finds user macro calls → performs `@macro.expand` → inlines expanded AST
3. Removes unused bindings
4. Returns self-contained AST

The build pipeline is composable:

```dvala
(entryAst) |> bundle({ target: "kmp" }) |> treeShake |> minify
```

## Plugin Model

Custom bundler plugins are macros. The bundler host provides `@macro.expand`, `@import.resolve`, and `@bundler.emitAsset` handlers.

## Selective Expansion

The bundler can selectively expand: eager-expand macros without runtime dependencies, leave runtime-dependent macros in the output for the runtime host to handle.

## Pre-expansion for Performance

The bundler can pre-expand user macros, promoting them to the zero-cost tier in bundled output. This means KMP never needs to see or expand these macros — it receives pure AST.

## Cross-Platform Bundles

- JS bundles for KMP: bundler pre-expands macros → KMP never sees them
- Continuation with macros crosses platforms: KMP handles `@macro.expand` on resume
- KMP host can delegate to JS for complex macros requiring the parser

## Implementation Plan

1. Bundler walks entry AST, resolves `import()` calls via `@import.resolve` effect
2. Inlines resolved module AST at import sites
3. Expands macro calls via `@macro.expand` effect
4. Tree-shaking: removes unused `let` bindings
5. Composable pipeline: each transform is a function/macro taking AST, returning AST
6. Tests: bundled output is self-contained, KMP can evaluate it

## Dependencies

- Macro system phases 1-6 (foundation, code templates, AST module, hygiene, pipe desugaring, core macros)
- `@import.resolve` effect
- `@bundler.emitAsset` effect (for asset handling)

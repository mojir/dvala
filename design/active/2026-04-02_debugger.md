# Dvala Debugger

**Status:** Draft
**Created:** 2026-04-02

---

## Overview

A debugger for Dvala, built incrementally in independent parts:

1. **Part 1 — Call Stack**: reconstruct human-readable call stacks from the continuation stack `k`. Standalone improvement — immediately gives stack traces on errors. No dependency on debug mode or `onNodeEval`.
2. **Part 2 — Preparation**: wire source positions and the `onNodeEval` hook into the evaluator, fix snapshot timing. (Shared with code coverage — see archived `2026-03-31_code-coverage.md`.)
3. **Part 3 — Core Debugger**: breakpoints, variable inspection, stepping, time travel.
4. **Part 4 — VS Code DAP Integration**: translate between DAP protocol and the core debugger.
5. **Part 5 — Playground Debug UI**: browser-based debug frontend for the playground, using the same core debugger (Part 3).

Parts can be implemented independently once their dependencies are met. Part 1 has no dependencies and should be done first. Parts 4 and 5 are independent frontends that both build on Part 3.

## Part 1 — Call Stack

### Motivation

Today, errors show only a single location (file:line:column + caret). No stack trace. This is the most impactful standalone improvement — every error immediately becomes easier to diagnose.

The same `reconstructCallStack()` function is then reused by the debugger (Part 3) and DAP adapter (Part 4) for stack trace display.

### Current state of frames

The continuation stack `k` is a `PersistentList<Frame>`. Each frame type carries different info:

| Frame Type | Has function name? | Has source position? |
|---|---|---|
| `EvalArgsFrame` | Yes (in AST node) | Yes (`sourceCodeInfo`) |
| `FnBodyFrame` | Yes (`UserDefinedFunction`) | Yes |
| `CallFnFrame` | **No** — name is lost | Yes |
| `AlgebraicHandleFrame` | Yes (`HandlerFunction`) | Yes |
| `HandlerClauseFrame` | Yes (`HandlerFunction`) | Yes |
| Structural frames (`SequenceFrame`, `IfBranchFrame`, `LetBindFrame`, etc.) | No — not calls | Yes (position only) |

### Required changes

1. **Store function name in `CallFnFrame`** — when pushing the frame, preserve the name (from the symbol or function object) so it's available for stack display

2. **Implement `reconstructCallStack(k: ContinuationStack): CallStackEntry[]`** — walks `k`, filters to call-relevant frames (`EvalArgsFrame`, `FnBodyFrame`, `CallFnFrame`, handler frames), extracts function name + source position for each

3. **`CallStackEntry` type:**
   ```typescript
   interface CallStackEntry {
     name: string              // function name or "<anonymous>"
     sourceCodeInfo?: SourceCodeInfo  // file, line, column
   }
   ```

4. **Include stack trace in `DvalaError`** — when throwing, capture the call stack from `k` and format it in the error message:
   ```
   TypeError: expected Obj
   myfile.dvala:42:15
   let x = foo(bar)
                  ^
     at foo        myfile.dvala:42:15
     at processAll utils.dvala:10:3
     at <main>     entry.dvala:5:1
   ```

### Implementation order

1. Add function name to `CallFnFrame`
2. Write `reconstructCallStack()` helper
3. Wire into `DvalaError` — format and display stack trace on errors
4. Add tests

## Part 2 — Preparation (shared with code coverage)

### 2a. Source Map in the Evaluator

`AstNode` is `[NodeType, Payload, nodeId: number]`. Source positions live in a separate `SourceMap` (`Map<nodeId, {start, end}>`) that is only built today when source text is passed to the parser (debug/error mode).

**Change**: when coverage or debug mode is active, always build the `SourceMap` and thread it through to the evaluator so the hook can resolve `node[2]` → `{line, col}`.

### 2b. `onNodeEval` Hook

```typescript
onNodeEval?: (node: AstNode, getContinuation: () => Continuation) => void | Promise<void>

interface Continuation {
  env: ContextStack           // current bindings
  k: ContinuationStack        // call stack
  resume: () => void          // continue execution
  getSnapshots: () => Snapshot[]  // prior post-effect snapshots (time travel)
}
```

`getContinuation()` is **lazy** — only called when needed. In the coverage path it is never called, costing nothing beyond the function call itself.

The hook returns `void | Promise<void>`. When a promise is returned, the evaluator awaits it before continuing — this is how the debugger pauses execution while waiting for DAP commands.

### 2c. Fix Snapshot Timing

Current behavior: snapshot taken **before** each effect.

New behavior:
1. **One snapshot at program start** — before the first node is evaluated
2. **One snapshot after each effect resolves** — effect result already in scope

**Why after?** A post-effect snapshot has the effect result baked in. Re-executing from it is pure and deterministic — no effect results need to be replayed. The segment between any two consecutive snapshots is always pure.

## Part 3 — Core Debugger

### Capabilities

| Capability | Mechanism |
|---|---|
| Continue | `resume()` from `getContinuation()` |
| Inspect variables | `env` from `getContinuation()` |
| Call stack display | `reconstructCallStack(k)` from Part 1 |
| Step over / into / out | host tracks depth across `onNodeEval` calls |
| Time travel (step back) | `getSnapshots()` → retrigger → pure re-execution to target node |
| Alter effect result / crash recovery | rewind to snapshot after effect N-1, re-handle effect N differently |

### Breakpoint Protocol

Host maintains a `Set<nodeId>` of breakpoints. On each `onNodeEval` call, host checks if `node[2]` is in the set. If yes, calls `getContinuation()` and holds `resume()` until "continue" is clicked.

### Time Travel

**Algorithm:**
```
rewind(targetNodeId):
  snapshot = latest snapshot with position ≤ target
  retrigger(snapshot)
    → pure re-execution, deterministic
    → onNodeEval stops when targetNodeId is reached
```

No effect history needed — pure re-execution is always correct. The segment between any two consecutive snapshots is always pure, since snapshots are taken after each effect resolves.

## Part 4 — VS Code DAP Integration

### Architecture

```
VS Code  ←—DAP (JSON over stdio)—→  Debug Adapter  ←—→  Dvala Runtime
                                     (Node process)       (onNodeEval hook)
```

The debug adapter is a Node.js process using `@vscode/debugadapter`. It translates between DAP's line-based protocol and Dvala's node-based internals. The core debugger logic (Parts 1–3) stays runtime-agnostic — the DAP adapter is purely a translation layer.

### DAP Request Mapping

| DAP Request | Dvala Implementation |
|---|---|
| `initialize` | report capabilities (supports step back, evaluate, etc.) |
| `launch` | parse + run Dvala program with `onNodeEval` hooked |
| `setBreakpoints` | translate `file:line` → `Set<nodeId>` via source map |
| `continue` | resolve the pending `onNodeEval` promise → `resume()` |
| `next` (step over) | set depth-based stop condition, resolve promise |
| `stepIn` / `stepOut` | same, different depth logic |
| `stepBack` | rewind to previous snapshot, re-execute to prior node |
| `stackTrace` | `reconstructCallStack(k)` from Part 1 |
| `scopes` / `variables` | walk `env` (ContextStack) from continuation |
| `evaluate` | eval expression in current env (debug console REPL) |
| `disconnect` | tear down runtime |

### Breakpoint Resolution

DAP breakpoints are `file:line`. The adapter translates these to `nodeId` via the source map:

1. Find all nodes whose start position is on the requested line
2. Pick the first (leftmost) node — this is the breakpoint target
3. Report the resolved line back to VS Code (DAP `Breakpoint.line`) so the dot snaps to the actual position

### Pause Mechanism

When `onNodeEval` hits a breakpoint (or step condition), it returns a `Promise<void>` that is held unresolved. The evaluator awaits this promise and pauses. When VS Code sends `continue` / `next` / `stepIn` / `stepOut`, the adapter resolves the promise and execution resumes.

### Time Travel via `stepBack`

DAP has `supportsStepBack` — advertise this in `initialize`. When VS Code sends `stepBack`:

1. Get snapshots from `getContinuation().getSnapshots()`
2. Pick the latest snapshot before the current node
3. Re-execute from that snapshot with `onNodeEval` stopping at the target node
4. Report the new stopped position back to VS Code

### Extension Packaging

The debug adapter lives in the existing VS Code extension (or a new `dvala-debug` extension). Key `package.json` contributions:

```json
{
  "contributes": {
    "debuggers": [{
      "type": "dvala",
      "label": "Dvala",
      "program": "./out/debugAdapter.js",
      "runtime": "node",
      "languages": ["dvala"],
      "configurationAttributes": {
        "launch": {
          "required": ["program"],
          "properties": {
            "program": {
              "type": "string",
              "description": "Path to .dvala file to debug"
            }
          }
        }
      }
    }]
  }
}
```

### Implementation Order

1. Minimal adapter: `launch`, `setBreakpoints`, `continue`, `disconnect`
2. Stepping: `next`, `stepIn`, `stepOut`
3. Inspection: `stackTrace`, `scopes`, `variables`, `evaluate`
4. Time travel: `stepBack`

## Part 5 — Playground Debug UI

A browser-based debug frontend for the playground. Uses the same core debugger (Part 3) — different UI, same `onNodeEval` + `getContinuation()` API.

### UI Elements

- **Breakpoint gutters** — click line numbers in the code editor to toggle breakpoints
- **Toolbar** — continue, step over, step in, step out, step back buttons
- **Call stack panel** — `reconstructCallStack(k)` from Part 1
- **Variables panel** — walk `env` from continuation, display bindings
- **Source highlighting** — highlight the current node being executed

### Architecture

```
Playground Editor  ←→  Debug UI Controller  ←→  Dvala Runtime (in-browser)
                                                  (onNodeEval hook)
```

Everything runs in the browser — no backend needed. The debug UI controller manages the same state a DAP adapter would (breakpoint set, current stop reason, pending resume) but drives a DOM-based UI instead of DAP messages.

### Pause Mechanism

Same as Part 4 — `onNodeEval` returns a `Promise<void>` that is held unresolved while the UI shows the paused state. Clicking "continue" or a step button resolves the promise.
